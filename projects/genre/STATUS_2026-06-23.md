### The headline finding (mk2.5 reproduced at scale)

mk3 added **10× the rows** (1,000 → 9,990 segments) and the result is a clean,
mechanism-backed **negative**: every reliability axis stays flat as a predictor of
held-out test rank.

```
Spearman(metric, TEST) at ×10 data (firewalled batch_refit):
   +0.429   CV accuracy (A)        ← still the only predictor
   +0.139   RRM (equal-weight)
   +0.130   fold-stability (−s)
   +0.099   RRM_w (deployed selector)
   −0.095   epistemic (−U)         ← still dead
```

**Why U could not move:** within-track segment correlation gives ICC = 0.681 →
design-effect 7.12 → 6,992 segments ≈ **982 independent-equivalent** (≈1.4×, not 10×).
The last-layer σ tightens only ~18% — inside the noise. Predicted *before* the run,
confirmed *by* the run. The mk2.5 boundary condition holds on a second dataset:
reliability-aware selection needs a *trained* Bayesian head (INFO 510), not post-hoc
LLLA on a frozen DET head. Refines, does not refute.

**Accuracy:** mk3 voted-to-track **0.807** vs mk2 **0.760** = **+4.7** (per-track,
apples-to-apples). Per-segment 0.789; vote gain +0.018 (segmentation denoising).
The +4.7 is real but bounded — exactly the ≈1.4× effective-N story.

## NEXT GOAL — integrate the three models + the attribution metric into FORGE

FORGE currently wraps **mk1 only**. The bundle contract is solid for expansion (all
bundles share the 8-file contract; `bundle.json` now self-declares capabilities).
Three pieces, roughly in order:

### 1. Multi-model in FORGE (mk1 · mk2 · mk3 as Model-panel dots)
- FORGE routes today: `/forge`, `/api/forge/{meta,posterior,datasweep,hmc}` (in `app/server.py`).
- These resolve ONE selected bundle. Generalize to take a `?model=<run_name>` (or read
  the sticky registry selection) so each tab can render any of `beardown`,
  `beardown_rrm`, `beardown_3sec`.
- Add the cfg-19 vs cfg-17 (and cfg-12) contrast as labeled dots — makes the selection
  pathology visible (the thesis story) in the dashboard.
- Mostly plumbing on the existing contract; LLLA/HMC/posterior already work per-bundle.

### 2. Wire the attribution metric (the new capability) — tabular bar chart first
- **Precompute (one-time, GPU box):** run the torch `dφ/dx` pass to write
  `attribution.npz`. The function is built and waiting:
  ```python
  from projects.genre.src import dataio, attribution
  d = dataio.load("fused3","stratified","projects/genre/data/raw",seed=42,
                  standardize=True,drop_length=True,image_dir="images_mel")
  attribution.precompute("projects/genre/models/beardown_3sec", d,
                         device="cuda", per_genre=25, tau_live=True)
  ```
  This writes `attribution.npz` and re-stamps `bundle.json` so its capabilities list
  gains `"attribution"`. (Do the same for mk1/mk2 if you want their bars too.)
- **Serve route:** add `/api/forge/attribution` reading
  `attribution.InputAttribution.from_cache(bundle_dir).payload(genre, tau, top)` →
  ranked named-feature bars (pure numpy, no torch in the serving path). The `dσ/dφ`
  closed form + τ-live recompute are already verified exact.
- **Tab:** new **capability-gated** "feature attribution" tab in `forge.html` — a bar
  chart over the 57 named tabular features, per-genre selectable, with the live-τ knob.
  Gate it on `bundle.json.capabilities` containing `"attribution"` (dark otherwise).
- **Thesis hook:** this *names* INFO 510 Table A3's latent features
  ("disco's epistemic uncertainty is driven by spectral_centroid_var…").
- Phase 2: the spectrogram saliency heatmap (image branch). Tabular bars first.
- Full design: `FORGE_input_feature_attribution_SPEC.txt` (Option 1, rung 2).

### 3. Self-describing tabs (the generic-FORGE spine)
- `bundle.json` already declares `capabilities`. Have each FORGE tab DECLARE its
  requirement and enable iff the bundle satisfies it (graceful degradation — a missing
  capability just leaves the tab dark). This is what makes "load any bundle, run FORGE"
  tractable later.
